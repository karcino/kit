Kit is a small software toolbox. Today it does two things — planning routes and touching the calendar — but the interesting question isn't what Kit does right now, it's what Kit is becoming. The answer: a service with one front door that every caller speaks to. A human in a terminal, a Python script, and an AI agent like Claude Code all reach the same engine — the agent via the Model Context Protocol. One core, no duplicated logic.

Once you have a single engine, you can put different kinds of mind in front of it. That is where the two agents come in. Same Kit core underneath; very different jobs.

The Orchestrator is a runtime agent. You give it an intent in plain language — "dinner at Luigi's at 19:30, get me there" — and it picks the right tools in the right order (kit_route, kit_cal_add, kit_plan_day) to make that intent true in the world: route calculated, calendar event created with a travel buffer, deep links attached. You reach it through whichever front door fits — the CLI, a Python import, Claude Code via MCP, or a POST /ask once the service is live. It is judged by latency and reliability — this is Kit's operational spine, and in a portfolio context its reliability under load is the demo: you are standing in a kitchen holding your phone, so seconds matter and mistakes are costly. Internally it runs a tool-use loop — an LLM emits structured tool calls, reads their results, decides what is next. A small model, one shot if possible, escalate only on evidence.

The Narrator is a design-time agent. It does not act in the world; it produces artifacts — architecture diagrams, concept pieces, spec drafts, this very page. To build them it reads the repo with Read, Glob, and Grep, then writes shaped markdown and HTML back into docs/. The trigger is not "do this for me" but "explain this, design this, document this". You invoke it through a Claude Code session — a plan file or a slash command — not a service endpoint. Internally it uses a reasoning-capable model in many turns: brainstorm into a plan, then plan into prose. The distinction between runtime and design-time is borrowed straight from programming-language semantics: the Orchestrator runs while the user has a need; the Narrator runs before, shaping what the system is. Latency is irrelevant here. A good Narrator output takes minutes and lives for months. What matters is craft, honesty, and legibility — which is why this page carries a "how this was made" link: a Narrator's product comes with its own provenance report attached.

The trick — and this is the bootstrap move — is that this very page is the first Narrator output. The agent whose architecture is being explained produced the explanation. Kit's design document and Kit's first proof-of-concept land together, in one artifact. The Narrator spec comes next, then the Orchestrator spec. But the thing that makes Kit a portfolio-worthy project — pitch-ready, Konstellation-ready starts with a page that can honestly explain itself.
